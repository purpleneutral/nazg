# Nexus Persistence Layer (`modules/nexus`)

Nexus is Nazg's durable memory. It stores project metadata, detector facts, filesystem snapshots, planner and
builder events, command history, and bot telemetry. The engine associates one `nexus::Store` with each runtime
and shares it across modules through the directive context.

---

## Quick Reference

**Initialize Store:**
```cpp
auto store = nexus::Store::create(db_path, log);
if (store && store->initialize()) { /* ready */ }
```

**Common Operations:**
```cpp
// Projects
int64_t id = store->ensure_project("/path/to/project");
auto proj = store->get_project(id);
store->update_project_metadata(id, "MyApp", "cpp", "cmake");

// Facts (detector results)
store->set_fact(id, "detector", "language", "cpp");
auto lang = store->get_fact(id, "detector", "language");

// Snapshots (change tracking)
store->add_snapshot(id, "abc123def", 42, 1024000);
auto snap = store->get_latest_snapshot(id);

// Events (decisions/outcomes)
store->add_event(id, "info", "planner", "Decision: BUILD", "Files changed");
auto events = store->get_events(id, 10);
```

**Database Location:**
- Default: `~/.local/state/nazg/nazg.db`
- Override: `[nexus] db_path` in config or `XDG_STATE_HOME`

---

## 1. Design Goals

- **Centralise state** – Keep long-lived knowledge (projects, facts, runs) in one SQLite database so Nazg can
  explain past decisions.
- **Accessible API** – Provide a high-level interface (`nexus::Store`) so modules avoid handwritten SQL.
- **Schema evolution** – Apply migrations automatically when new tables are added.
- **User-owned data** – Store the database under XDG locations by default and respect config overrides.

---

## 2. Architecture Overview

| Component | Responsibility |
|-----------|----------------|
| `nexus::Store` | Public facade exposing project, fact, snapshot, event, command, and bot APIs. |
| `nexus::Connection` | Backend abstraction (`sqlite` implementation ships today). |
| `nexus::Migrator` | Executes SQL migrations from `modules/nexus/migrations`. |
| `nexus::Config` | Reads `[nexus]` settings from `config::store` and applies XDG defaults. |

`engine::runtime::init_nexus()` calls `Store::create(path, logger)` and then `initialize()`; if either step fails
the engine logs an error and skips further database work.

---

## 3. Schema Snapshot

Current tables (see `modules/nexus/migrations/001_init.sql` and later migrations):

| Table | Purpose |
|-------|---------|
| `projects` | Known project roots with metadata (path, display name, primary language, build system). |
| `facts` | Namespaced key/value store for detector output and other module facts. |
| `snapshots` | Tree hash history with counts and byte totals. |
| `events` | Structured log of planner decisions, builder outcomes, and other noteworthy events. |
| `commands` | History of executed commands (plan command, args, exit code, duration). |
| `bot_hosts` | Registered remote hosts for bot runs (label, address, last status). |
| `bot_runs` | Individual bot executions (bot name, host, status, durations). |
| `bot_reports` | JSON payloads generated by bots. |
| `migrations` | Versioning table used by the migrator.

The schema is additive; migrations should never destructively alter existing tables without a compatibility path.

## 4. Key APIs

`modules/nexus/include/nexus/store.hpp` exposes the primary surface. Highlights include:

```cpp
auto store = nexus::Store::create(db_path, log);
if (store && store->initialize()) { /* ready */ }
```

| Method | Description |
|--------|-------------|
| `int64_t ensure_project(const std::string& root_path)` | Insert or fetch a project row (by canonical path). |
| `std::optional<Project> get_project(int64_t id)` | Retrieve project metadata. |
| `std::optional<Project> get_project_by_path(const std::string&)` | Lookup by root path. |
| `void update_project(int64_t id, const Project&)` | Write back metadata updates (name, language, build system). |
| `void set_fact(int64_t project_id, const std::string& ns, const std::string& key, const std::string& value)` | Upsert facts. |
| `std::optional<std::string> get_fact(...)` | Fetch a single fact value. |
| `std::map<std::string, std::string> list_facts(project_id, ns)` | Retrieve all facts for a namespace. |
| `int64_t add_snapshot(project_id, hash, file_count, total_bytes)` | Persist snapshot results. |
| `std::optional<Snapshot> latest_snapshot(project_id)` | Get the most recent snapshot. |
| `void add_event(project_id, level, tag, message, metadata)` | Log events for assistant/diagnostics. |
| `std::vector<Event> recent_events(project_id, limit)` | Query recent events. |
| `void record_command(project_id, command, args, exit_code, duration_ms)` | Track executed commands. |
| `std::vector<CommandRecord> recent_commands(...)` | Fetch historic commands (global or per project). |
| `void prune_events(project_id, keep)` / `prune_commands(keep)` | Trim history to a bounded size. |
| `int64_t add_bot_host(label, address, ssh_config)` | Register remote targets for bots. |
| `void update_bot_host_status(host_id, status)` | Store latest health string. |
| `std::vector<std::map<std::string,std::string>> list_bot_hosts()` | Inspect bot host registry. |
| `int64_t begin_bot_run(bot, host_id)` / `finish_bot_run(run_id, status, exit_code, duration_ms)` | Track bot executions. |
| `void add_bot_report(run_id, json_payload)` | Persist structured results from bots. |
| `std::optional<std::string> latest_bot_report(bot, host_id)` | Retrieve the newest report for display. |

This API surface continues to expand; consult the header for the full list and signatures.

## 5. Usage Patterns

### Detector
```cpp
auto project_id = store->ensure_project(cwd);
store->set_fact(project_id, "detector", "language", info.language);
```

### Snapshotter
```cpp
auto snapshot_id = store->add_snapshot(project_id, result.tree_hash,
                                        result.file_count, result.total_bytes);
```

### Planner & Builder
```cpp
store->add_event(project_id, "info", "planner", "Decision: BUILD", plan.reason);
store->record_command(project_id, plan.command, plan.args, result.exit_code, result.duration_ms);
```

### Bots
```cpp
auto host_id = store->add_bot_host("lab", "dev@lab-box", "~/.ssh/config");
auto run_id = store->begin_bot_run("doctor", host_id);
// … execute bot …
store->add_bot_report(run_id, payload_json);
store->finish_bot_run(run_id, "warning", exit_code, duration_ms);
```

## 6. Configuration & Paths

Default location: `~/.local/state/nazg/nazg.db` (or `$XDG_STATE_HOME/nazg/nazg.db`). Override via config:

```toml
[nexus]
db_path = "/var/lib/nazg/nexus.db"
```

Ensure the user running Nazg has read/write access. WAL mode is enabled automatically to improve concurrency
and crash resilience.

---

## 7. Migration Workflow

1. Add a numbered SQL file to `modules/nexus/migrations/` (e.g., `002_bot_tables.sql`).
2. Update the migrator version constant if required.
3. Extend `Store` with helper methods for the new tables.
4. Document the schema change (this file or module-specific doc).
5. Run Nazg; `initialize()` will detect and apply pending migrations.

Migrations are applied sequentially; ensure each file is idempotent and safe to rerun.

---

## 8. Diagnostics & Maintenance

- All major operations log success/failure when a logger is supplied. Run `nazg --verbose` to inspect database
  messages on the console.
- Use `store->prune_events` and `store->prune_commands` to limit history size if the database grows quickly.
- For manual inspection, open the database with `sqlite3 ~/.local/state/nazg/nazg.db` and explore tables.

---

## 9. Troubleshooting

| Symptom | Likely Cause | Remedy |
|---------|--------------|--------|
| `Failed to create database` | Parent directory missing or unwritable | Adjust `[nexus] db_path` or fix directory permissions. |
| `Migration failed` | Invalid SQL or missing file | Check logs for the failing migration and verify the script. |
| Facts not persisting | `ensure_project` not called before `set_fact` | Always ensure the project row exists. |
| `database is locked` errors | Concurrent access from multiple Nazg instances or other tools | WAL mode reduces contention; close other connections or serialise access. |

Nexus gives Nazg a memory. Keep migrations tidy, use the helper APIs, and the assistant will always have the
context it needs.
